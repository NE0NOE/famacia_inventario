                        1. INTRODUCCIÓN
================================================================================

1.1 Descripción del Proyecto
-----------------------------
El proyecto "Farmacia Dulce Esperanza" es un sistema de gestión integral
diseñado para modernizar y automatizar las operaciones diarias de una farmacia.
La solución abarca el control de inventario, la gestión de ventas, la
administración de clientes y proveedores, y la generación de facturas.

El sistema fue desarrollado como una solución multiplataforma, es decir,
funciona en tres plataformas distintas:

  - WEB: Un panel administrativo accesible desde cualquier navegador.
  - MÓVIL (ANDROID): Una aplicación nativa instalable en celulares/tablets.
  - ESCRITORIO (LINUX): Una aplicación de escritorio instalable en computadoras.

Las tres plataformas se conectan al mismo servidor backend (la misma base de
datos), lo que garantiza que la información esté siempre sincronizada.


1.2 Problema que resuelve
--------------------------
Muchas farmacias pequeñas y medianas aún manejan sus operaciones de forma
manual (cuadernos, hojas de cálculo), lo que genera:

  - Errores en el control de stock (productos vencidos, faltantes no detectados).
  - Lentitud en el cobro de ventas (cálculos manuales de totales e impuestos).
  - Falta de información sobre clientes frecuentes.
  - Dificultad para llevar un registro de proveedores y compras.
  - Imposibilidad de generar reportes de ventas para toma de decisiones.

Este sistema resuelve todos estos problemas al digitalizar cada proceso.


1.3 Objetivo General
---------------------
Desarrollar un sistema de gestión integral para farmacia que permita
administrar inventario, ventas, clientes y proveedores desde múltiples
plataformas (web, móvil y escritorio), mejorando la eficiencia operativa
y la toma de decisiones del negocio.


1.4 Objetivos Específicos
---------------------------
  - Implementar un módulo de inventario con alertas de stock bajo.
  - Desarrollar un Punto de Venta (POS) ágil con cálculo automático.
  - Crear un sistema de registro y fidelización de clientes (puntos).
  - Integrar un módulo de proveedores y órdenes de compra.
  - Generar facturas digitales y reportes de ventas.
  - Desplegar el sistema en un servidor accesible por dominio propio.
  - Generar compilados instalables (APK para Android, AppImage para Linux).


1.5 Tecnologías Utilizadas
----------------------------
Se seleccionó un stack tecnológico moderno y robusto:

  BACKEND (API del servidor):
  ├── Node.js          → Plataforma de ejecución JavaScript del lado del servidor.
  ├── Express.js       → Framework web para crear la API RESTful.
  ├── PostgreSQL       → Base de datos relacional (almacena toda la info).
  ├── Cors             → Middleware de seguridad para peticiones entre dominios.
  └── Dotenv           → Manejo seguro de variables de entorno (contraseñas, etc).

  FRONTEND WEB (Panel Administrativo):
  ├── React.js         → Biblioteca para construir interfaces de usuario interactivas.
  ├── Vite             → Empaquetador de módulos ultrarrápido (reemplaza a Webpack).
  ├── Tailwind CSS     → Framework de estilos CSS basado en utilidades.
  ├── Lucide React     → Biblioteca de iconos SVG consistente.
  └── React Router     → Navegación SPA (Single Page Application) sin recargar página.

  APLICACIÓN MÓVIL (Android):
  ├── React.js + Vite  → Misma tecnología que la web (reutilización de código).
  ├── Capacitor        → Puente que convierte la app web en app nativa Android.
  └── Conexión al servidor vía dominio público (dulcesperanzafm.dpdns.org).

  APLICACIÓN DE ESCRITORIO (Linux):
  └── Electron         → Framework que permite ejecutar apps web como apps de escritorio.

  DESPLIEGUE:
  ├── Docker           → Contenedores para empaquetar y ejecutar la aplicación.
  ├── Docker Compose   → Orquestador que levanta todos los servicios juntos.
  ├── Nginx            → Servidor web que sirve el frontend en producción.
  └── Cloudflare       → DNS y dominio personalizado (dulcesperanzafm.dpdns.org).

                    2. METODOLOGÍA DEL PROYECTO
================================================================================

2.1 Metodología de Desarrollo
-------------------------------
Se utilizó una metodología ágil incremental, donde el sistema se fue
construyendo por módulos funcionales que se integraban paulatinamente.
La dinámica de trabajo fue la siguiente:

  1. Se definieron los requerimientos funcionales (qué debe hacer el sistema).
  2. Se diseñó la base de datos con todas las tablas necesarias.
  3. Se desarrolló el backend (API) primero, ya que es el núcleo del sistema.
  4. Se construyó el frontend web conectándolo al backend.
  5. Se adaptó el frontend para la versión móvil y escritorio.
  6. Se realizaron pruebas de integración y corrección de errores.
  7. Se desplegó en un servidor real con Docker.

Cada módulo pasó por el ciclo: Diseño → Codificación → Prueba → Integración.


2.2 Herramientas de Desarrollo
---------------------------------
  - Editor de código: Visual Studio Code (VS Code).
  - Control de versiones: Git + GitHub.
  - Base de datos: PostgreSQL (local en desarrollo, Docker en producción).
  - Pruebas de API: Navegador / herramientas de red del navegador.
  - Compilación móvil: Android Studio + Capacitor CLI.
  - Compilación escritorio: Electron Builder.
  - Despliegue: Docker + Docker Compose en servidor Ubuntu.

3.2 Descripción Detallada de Cada Módulo
------------------------------------------

A. MÓDULO DE LOGIN / AUTENTICACIÓN
   Archivo backend: routes/auth.js
   Páginas: Login.jsx (web y móvil)

   - El sistema requiere usuario y contraseña para acceder.
   - Las contraseñas se almacenan como hash (cifradas) en la base de datos.
   - Existen dos roles: "admin" (acceso total) y "seller" (acceso limitado).
   - Al iniciar sesión, se guarda la sesión del usuario.
   - Credenciales por defecto: usuario "admin", contraseña "admin123".

B. MÓDULO DE INVENTARIO (PRODUCTOS)
   Archivo backend: routes/products.js
   Páginas: Inventory.jsx (web y móvil)

   - Permite agregar, editar, eliminar y buscar productos.
   - Cada producto tiene: SKU (código único), nombre, categoría, precio, stock.
   - El stock se actualiza automáticamente al registrar ventas o compras.
   - Se muestran alertas cuando el stock de un producto está bajo.

C. MÓDULO DE PROVEEDORES
   Archivo backend: routes/suppliers.js
   Páginas: Suppliers.jsx (web y móvil)

   - Registro completo de proveedores con datos de contacto.
   - Gestión de órdenes de compra hacia proveedores.
   - Cada orden de compra puede tener múltiples productos.
   - Al recibir una orden, el stock se actualiza automáticamente.

D. MÓDULO DE VENTAS / PUNTO DE VENTA (POS)
   Archivo backend: routes/sales.js
   Páginas: POS.jsx (web y móvil)

   - Interfaz de caja estilo punto de venta profesional.
   - Se buscan productos y se agregan al carrito de venta.
   - Cálculo automático de subtotal, impuestos y total.
   - Métodos de pago: Efectivo y Tarjeta.
   - Al confirmar la venta, se descuenta del stock automáticamente.

E. MÓDULO DE CLIENTES
   Archivo backend: routes/clients.js
   Páginas: Clients.jsx (web y móvil)

   - Registro de clientes con cédula/RUC, nombre, teléfono, email.
   - Sistema de puntos de fidelidad (se acumulan puntos por cada compra).
   - Registro de última visita del cliente.
   - Búsqueda rápida de clientes por nombre o cédula.

F. MÓDULO DE FACTURACIÓN
   Páginas: Invoicing.jsx (web y móvil)

   - Generación de facturas digitales por cada venta realizada.
   - Historial completo de transacciones con filtros.
   - Detalle de cada factura: productos, cantidades, precios, totales.

G. MÓDULO DE DASHBOARD (PANEL PRINCIPAL)
   Páginas: Dashboard.jsx (web y móvil)

   - Vista general con estadísticas del negocio.
   - Ventas del día, semana y mes.
   - Productos con stock bajo (alertas).
   - Resumen de clientes y proveedores.

H. MÓDULO DE CONFIGURACIÓN
   Páginas: Settings.jsx (web y móvil)

   - Gestión de usuarios del sistema.
   - Configuraciones generales del sistema.


3.3 Base de Datos (PostgreSQL)
---------------------------------
El sistema utiliza PostgreSQL como sistema de gestión de base de datos.
El esquema tiene 8 tablas con las siguientes relaciones:

  products ←──── sale_items ────→ sales ────→ clients
  products ←──── purchase_items ─→ purchases ─→ suppliers
  users (independiente)

  Relaciones:
  - Una VENTA (sales) puede tener muchos ITEMS (sale_items).         [1:N]
  - Cada ITEM de venta referencia a un PRODUCTO (products).          [N:1]
  - Una VENTA puede estar asociada a un CLIENTE (clients).           [N:1]
  - Una COMPRA (purchases) puede tener muchos ITEMS (purchase_items). [1:N]
  - Cada ITEM de compra referencia a un PRODUCTO (products).         [N:1]
  - Una COMPRA está asociada a un PROVEEDOR (suppliers).             [N:1]

  Docker Compose levanta 3 contenedores:
  ┌─────────────────────────────────────────────┐
  │  1. farma_postgres  (Puerto 5432)           │
  │     → Base de datos PostgreSQL 15           │
  │     → Volumen persistente para los datos    │
  │                                             │
  │  2. farma_backend   (Puerto 3000)           │
  │     → API Node.js/Express                   │
  │     → Se conecta a PostgreSQL               │
  │                                             │
  │  3. farma_frontend  (Puerto 80)             │
  │     → Frontend React compilado              │
  │     → Servido por Nginx                     │
  └─────────────────────────────────────────────┘

  Comando para levantar todo:
    docker compose up --build -d

  Dominio configurado: dulcesperanzafm.dpdns.org (Cloudflare DNS)


3.6 Compilados Generados
--------------------------
Se generaron dos tipos de compilados distribuibles:

  A. APK para Android (app-debug.apk):
     - Se genera con: npm run build → npx cap sync android → Gradle.
     - Se instala directamente en dispositivos Android.
     - Requiere Android 7.0 o superior.
     - Se conecta al backend vía dominio público (no requiere misma WiFi).

  B. AppImage para Linux (Farmacia Dulce Esperanza-1.0.0.AppImage):
     - Se genera con: npm run build:linux (Electron Builder).
     - Es un ejecutable portable (no requiere instalación).
     - Solo se necesita dar permisos de ejecución y hacer doble clic.

  Ambos están disponibles para descarga en:
  https://github.com/NE0NOE/famacia_inventario/releases/tag/v1.0.0

                       5. RECOMENDACIONES

ANEXO A: COMANDOS DE EJECUCIÓN DEL PROYECTO
---------------------------------------------
Para ejecutar el proyecto en modo desarrollo, se necesitan 2 terminales:

  Terminal 1 - Backend:
    cd farma-backend
    npm install
    npm run dev

  Terminal 2 - Frontend Web:
    cd farma-web
    npm install
    npm run dev

  Acceder en el navegador: http://localhost:5173
  Credenciales: admin / admin123


ANEXO B: COMANDOS DE COMPILACIÓN
----------------------------------

  Compilar Frontend Web (archivos estáticos):
    cd farma-web
    npm run build
    # Salida: farma-web/dist/

  Compilar APK Android:
    cd farma-mobile
    npm run build
    npx cap sync android
    cd android
    ./gradlew assembleDebug
    # Salida: android/app/build/outputs/apk/debug/app-debug.apk

  Compilar Escritorio Linux:
    cd farma-desktop
    npm install
    npm run build:linux
    # Salida: farma-desktop/dist/*.AppImage


ANEXO C: DESPLIEGUE CON DOCKER
---------------------------------

  Levantar todos los servicios:
    cd farma     (carpeta raíz)
    docker compose up --build -d

  Ver logs:
    docker compose logs -f

  Detener servicios:
    docker compose down

  Verificar que los contenedores estén corriendo:
    docker ps


ANEXO D: VARIABLES DE ENTORNO REQUERIDAS
-------------------------------------------
El proyecto requiere las siguientes variables de entorno:

  Para el backend (.env en farma-backend/):
    PORT=3000
    DB_USER=postgres
    DB_HOST=localhost        (o "postgres" si usa Docker)
    DB_NAME=farma_db
    DB_PASSWORD=tu_contraseña
    DB_PORT=5432

  Para Docker (.env en la raíz farma/):
    DB_PASSWORD=tu_contraseña

  Para la app móvil (.env en farma-mobile/):
    VITE_API_BASE_URL=http://dulcesperanzafm.dpdns.org/api
    (Esta variable apunta al dominio público. Nginx redirige /api/ al backend.)
    (Alternativa para desarrollo local: http://localhost:3000/api)
